# Precompile transform

We've ported Deno's precompile to Vite, making precompilation available on
any JS runtime — Node, Bun, Deno, Edge, browser.

`@vincle/vite-plugin-precompile` transforms JSX at build time. Instead of
emitting `jsx()` calls that create intermediate objects, it rewrites JSX into
tagged template literals — less allocations, less GC pressure, faster
rendering.

```
// Before: runtime creates a VNode tree
jsx("div", { class: "x" }, jsx("span", {}, "hello"))

// After (precompile): direct string assembly
jsxTemplate`<div ${jsxAttr("class", "x")}><span>${jsxEscape("hello")}</span></div>`
```

The plugin auto-detects your `jsxImportSource` and wires the helpers through a
virtual module (`virtual:vincle-precompile-runtime`) — no physical adapter
file is needed. The three helpers (`jsxTemplate`, `jsxAttr`, `jsxEscape`) are
provided by a [**Deno-precompile-compatible**](https://docs.deno.com/runtime/reference/jsx/)
runtime – Preact, Hono, or @vincle/core.

## Compatible JSX frameworks

| Framework | `jsxImportSource` | Compatible |
| --------- | ----------------- | ---------- |
| Vincle    | `@vincle/core`    | ✅         |
| Preact    | `preact`          | ✅         |
| Hono      | `hono/jsx`        | ✅         |
| React     | `react`           | ❌         |

React does not export the `jsxTemplate` helper that the precompile transform
relies on. If your project targets React, the plugin throws a build error.

## Two layers of independence

| Layer                      | What it means                                                                                                        |
| -------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| **JS runtime agnostic**    | The helpers are pure functions — no platform imports. The generated output runs on Node, Bun, Deno, Edge, browser.   |
| **JSX framework agnostic** | The plugin auto-detects your runtime and wires the helpers. Works with Preact, Hono, Vincle — no adapter, no config. |

## Usage

Just add the plugin:

```ts
import precompile from "@vincle/vite-plugin-precompile";
import { defineConfig } from "vite";

export default defineConfig({
  plugins: [precompile()],
});
```

No adapter file, no extra config. The plugin detects the runtime from your
`jsxImportSource` and wires up the helpers automatically through a virtual module.

### Custom runtime

If you want to use your own implementation of the three helpers, point
`runtimeSource` to your module:

```ts
precompile({
  runtimeSource: "custom/jsx-runtime",
});
```

## Build-time sanitization (secure mode)

**Enabled by default.** Static attribute values are hardened at build time
using the runtime's own attribute serializer (`jsxAttr`). This catches XSS
vectors that are invisible to type-checking — a literal `href="javascript:…"`
in your source passes any linter but is a real vulnerability:

```ts
// Static string — looks harmless, linter passes
<a href="javascript:alert(1)">click</a>

// Without secure mode → inlined verbatim in the bundle
//   <a href="javascript:alert(1)">click</a>

// Secure mode → blocked at build time
//   <a href="#blocked">click</a>
```

The same URL-scheme blocking, unsafe-CSS dropping, and name remapping that
the runtime applies to _dynamic_ values also applies to static ones —
but resolved before the browser runs a single byte.

### How it works

The plugin loads `jsxAttr` from your runtime module at build time and runs
every static attribute value through it. The serialized result is inlined
directly into the generated template literal, so **there is zero runtime
cost** — no wrapper objects, no deferred calls, no extra branching.

```ts
// vite.config.ts — secure mode is ON by default
import precompile from "@vincle/vite-plugin-precompile";
import { defineConfig } from "vite";

export default defineConfig({
  plugins: [precompile()],
});
```

### What is caught?

| Attack surface        | Example                                   | Secure mode output        |
| --------------------- | ----------------------------------------- | ------------------------- |
| `javascript:` URLs    | `<a href="javascript:alert(1)">x</a>`     | `<a href="#blocked">x`    |
| `vbscript:` URLs      | `<a href="vbscript:msgbox">x</a>`         | `<a href="#blocked">x`    |
| Unsafe `data:` URLs   | `<iframe src="data:text/html,<script>…">` | `<iframe src="#blocked">` |
| `on*` event handlers  | `<button onclick="alert(1)">x</button>`   | `<button>x</button>`      |
| React names in output | `<div className="box">x</div>`            | `<div class="box">x`      |

### Opting out: Deno-precompile-compatible mode

Set `secure: false` to match Deno's own `jsx: "precompile"` transform:
static attributes are trusted and inlined verbatim (name-remapped and
HTML-escaped, but no URL/CSS sanitization applied at build time).

```ts
precompile({
  secure: false,
});
```

Use this when migrating an existing Deno precompile setup, when you have
your own build-time sanitizer, or when the build-time dependency on the
runtime module is undesirable.

Build-time sanitization works with any runtime module that exports
`jsxAttr` — Vincle, Preact, and Hono all support it.
