# FAQ

## What is Vincle and when should I use it?

Vincle is a JSX-to-HTML renderer designed exclusively for server-side use.
Use it whenever the output you want is an HTML string:

- **Static sites** — render pages to `.html` files at build time
- **Email templates** — compose rich HTML without a template language
- **SSR responses** — serve HTML from any HTTP framework
- **API responses** — return HTML fragments from an endpoint
- **PDF sources** — generate HTML to feed into a headless browser

Vincle is **not** for client-side rendering or hydration. If you need
interactive components in the browser, pair it with HTMX, Turbo, Alpine.js,
or a dedicated frontend framework.

## How is Vincle different from React?

React's renderers carry a virtual DOM, a reconciler, and a hydration
contract. Vincle is built exclusively for server-side HTML generation:

|                | Vincle                              | react-dom/server |
| -------------- | ----------------------------------- | ---------------- |
| Bundle size    | ~3 KB gzip                          | ~140 KB          |
| Dependencies   | 0                                   | React runtime    |
| Async          | Native `await`                      | Suspense-only    |
| Virtual DOM    | None                                | Full             |
| Client runtime | None                                | Hydration        |
| Security       | Escape + URL block + handler filter | Escape only      |

## Can I use Vincle with React components?

No. Vincle uses its own [JSX runtime](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Operators/JSX)
and does not support React hooks, refs, or the React ecosystem. The syntax
is the same — the runtime is incompatible.

If you need React ecosystem libraries (MUI, Radix, React Router, TanStack),
use React's own server renderer. Vincle is for projects where you control
the component tree and don't need client-side interactivity.

## What runtimes does Vincle support?

| Runtime                | Support | Notes                               |
| ---------------------- | ------- | ----------------------------------- |
| **Node.js 20+**        | ✅ Full | Native `AsyncLocalStorage`          |
| **Bun**                | ✅ Full | Node-compatible `AsyncLocalStorage` |
| **Deno**               | ✅ Full | Node-compatible `AsyncLocalStorage` |
| **Cloudflare Workers** | ✅ Full | Requires `nodejs_compat_v2` flag    |

No DOM shim, no polyfill, no browser build step. Vincle works wherever
JavaScript runs.

## How do I handle async data fetching?

Any component can be `async`. `renderToString` walks the tree depth-first
and awaits every async component it encounters:

```tsx
async function Page() {
  const user = await db.users.find(id);
  const posts = await db.posts.findByUser(id);
  return (
    <html>
      <body>
        <h1>{user.name}</h1>
        {posts.map((post) => (
          <Post key={post.id} {...post} />
        ))}
      </body>
    </html>
  );
}

const html = await renderToString(<Page id="42" />);
```

Sibling async branches run concurrently. The returned `Promise<string>`
resolves when every branch has finished rendering. No prop-drilling, no
top-level loader — the component that needs data is the component that
fetches it.

## How does security work in Vincle?

Security is the default, not an opt-in. Every value embedded in JSX is
automatically sanitised:

- **Text content** — `&`, `<`, `>` escaped
- **Attribute values** — also `"` escaped
- **URL attributes** — `javascript:`, `vbscript:`, `data:` blocked
- **Event handlers** — functions silently dropped
- **CSS values** — `expression()`, `javascript:` blocked

The only escape hatch is `raw()`, which is explicit and opt-in. See the
[Security model](/guide/security) for the full details.

## Can I use Vincle for streaming HTML?

Yes. The [`@vincle/flow`](/integration/overview) package adds deferred
fragments and streaming. The shell is sent immediately while slow
components render concurrently. Five adapters let you choose your
client mechanism:

- **NativeAdapter** — WICG spec with a ~550 B polyfill
- **TurboAdapter** — Hotwire Turbo Streams
- **HtmxAdapter** — HTMX OOB swaps
- **WebPlatformAdapter** — native browser support, no JS
- **EsiAdapter** — CDN-level ESI composition

All adapters work in both streaming and static generation modes.

## What is the bundle size?

| Package        | Size (gzip) | Dependencies        |
| -------------- | ----------- | ------------------- |
| `@vincle/core` | ~3 KB       | 0                   |
| `@vincle/flow` | ~2 KB       | 0 (depends on core) |

No hidden dependencies, no runtime you don't need. Every byte is accounted
for.

## How do I debug rendering issues?

Vincle logs warnings for:

- **Invalid tag names** — element omitted from output
- **Dropped event handlers** — function values on `on*` props
- **Missing context values** — `useContext` called before `setContext`
- **Invalid URL schemes** — blocked and replaced with `#blocked`

TypeScript catches type errors at build time. For runtime issues in
streaming, `@vincle/flow` provides `onError` handlers for per-fragment
error recovery:

```tsx
serve(req, () => <Page />, NativeAdapter, {
  onError: (error, { id, kind }) => {
    console.error(`Fragment ${id} failed:`, error);
    return <p>This section failed to load.</p>;
  },
});
```
